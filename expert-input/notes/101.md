## Summary

The expert describes how trade-ins — taking a customer's vehicle in part-exchange for one in the workshop's stock — should be handled. The recommendation is to mirror how Danish OE systems (Dracar/Keyloop, Vitec Datamann, CAC) already do it: the trade-in is created in **Draft** first, then **attached to the sales invoice of the outgoing vehicle**, and a single invoice posting both completes the sale of the outgoing vehicle and takes the trade-in into stock. The invoice itself carries information for both vehicles, and the cashbook posting reflects the trade-in value as a deduction against the customer's amount due.

## Points

1. **Reason for change.** Current trade-in handling needs to be revised. The expert has reviewed how this is done in OE systems used by Danish workshops (Dracar / Keyloop and Vitec Datamann) and notes they all match how **CAC** does it. Vehicle Trade V2 should follow the same pattern.

2. **Reference systems.** Dracar / Keyloop and Vitec Datamann (Danish dealer-management systems). CAC is treated as the reference implementation of the flow.

3. **Step 1 — outgoing vehicle in stock.** The workshop already holds the outgoing vehicle in stock — the one the customer wants to buy. Example used in the document: GPL002 / EQ1. This step already works in the current system.

4. **Step 2 — the customer's trade-in offer.** A customer wants the outgoing vehicle but offers another vehicle (a Fiat 500 in the example) as part-payment. The Fiat must enter the workshop's stock as part of the sale.

5. **Step 3 — create the trade-in as Draft.** The trade-in vehicle is added to the system as a **Draft** equipment row (with whatever vehicle-identity and agreed-value information is needed).

6. **Step 4 — initiate sale of the outgoing vehicle.** On the outgoing vehicle (TD53604 in the example), the user presses **"+ Faktura"** ("+ Invoice") to start the sales invoice.

7. **Step 5 — attach the trade-in on the invoice screen.** On the sales-invoice page there must be a UI affordance to **select / attach** the existing Draft trade-in. The expert suggests a few possible locations:
   - a tab,
   - a section below the existing invoice content, or
   - an entry under an **Actions** menu — *"or in a better way if you have this"*.

   The expert is explicit that the placement is open; the requirement is that the affordance exists.

8. **Step 6 — post the invoice. One posting does two things atomically:**
   - The outgoing vehicle is sold to the customer (status `Stock → Sold`).
   - The trade-in vehicle is taken into stock at the agreed value: status `Draft → Stock`, lot created, inventory aggregate incremented.

9. **Single document, both vehicles.** The resulting sales invoice carries **all information about the trade-in vehicle** as well, not just the outgoing one — the invoice is the audit document for both sides of the exchange.

10. **Accounting entries (e-conomic).** The expert illustrates with two postings:
    - **Cashbook:** the trade-in vehicle is added to inventory at its agreed value. The expert's screenshot shows this as a **negative** cash entry (example: −11 000 DKK) representing the value the workshop has effectively "paid" for the trade-in.
    - **Customer invoice:** the trade-in value is **deducted** from the cash amount the customer owes.
    - **Net effect:** the workshop's cash receipt = outgoing sale price − trade-in value.

## Ambiguities

- **Trade-in's purchase type.** The example involves a private customer, which implies the trade-in falls under the **margin scheme** (purchase type 2 — bought from a private citizen via *afregningsbilag*). The document doesn't say whether the system should default to that or let the user pick (the buyer might be a business, in which case the trade-in is a purchase invoice with VAT).
- **Trade-in valuation entry point.** Where the agreed trade-in value is entered — on the Draft row at creation, or on the sale screen when attaching — is not specified.
- **Linked-document cancellation.** If the sales invoice is later credited (`Sold → Stock`), what happens to the trade-in? Does it stay in stock as a normal purchase, or does it revert (`Stock → Reverted`)? Not addressed.
- **"+ Faktura" vs the system's existing sale flow.** The expert's screenshot suggests the sale starts from the outgoing equipment row by pressing a "+ Faktura" button. Whether this matches the current sale UI path in V1 isn't stated.
- **Existing "draft purchase document" requirement.** `docs/30-domain-model/equipment.md` says of the `(new) → Draft` transition that *"a draft purchase document is normally created alongside"*. For a trade-in there is no separate purchase document — the sales invoice for the outgoing vehicle serves that role. The current spec wording therefore needs adjustment for trade-ins.
