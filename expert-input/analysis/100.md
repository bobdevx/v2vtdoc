# Analysis 100 — Vehicle taxes, customs, and export refunds on the Danish vehicle trade

Source: [notes/100.md](../notes/100.md)

## Per-point findings

### Point 1 — Danish vehicle taxes are uniquely high; model *registreringsafgift* explicitly

- **Current spec:** `docs/40-purchase-types/README.md` mentions *registreringsafgift* under purchase types 3 (EU import) and 4 (non-EU import) — "due on first Danish registration… not VAT and is not deductible". `docs/60-accounting/README.md` lists *Treatment of registreringsafgift in the chart of accounts — separate pass-through account vs absorbed into inventory cost* as an open question. `docs/40-purchase-types/README.md` also lists *Treatment of registreringsafgift — in carrying value vs as a pass-through* as open.
- **Expert input:** *Registreringsafgift* is up to 200–300 % of vehicle value; it must be a first-class concept in the model, not just a footnote.
- **Verdict:** *confirms* and *extends* — the open questions in the current spec are now answered by the expert's proposed JobType/SKU mechanism.
- **Recommended action:** Resolve the two open questions in favour of the expert's approach — a **dedicated JobType / SKU pair** for the registration-tax outlay (treated as pass-through on resale, no VAT) — and document it in `docs/30-domain-model/job-card.md` and `docs/30-domain-model/product.md`. Update `docs/40-purchase-types/README.md` to reference that mechanism for the import types.

### Point 2 — Every e-conomic posting carries the unique vehicle number

- **Current spec:** `docs/30-domain-model/equipment.md` defines the Equipment code (`EQ######`) as the per-vehicle identifier. `docs/60-accounting/README.md` says the integration is "API-based, asynchronous, and idempotent" and that each X2 document "produces a posting batch identified by a stable external reference" — but does not explicitly require the equipment code to be on every line.
- **Expert input:** Every e-conomic transaction must be marked with the unique vehicle number.
- **Verdict:** *extends* — the existing spec implies traceability but does not state it as a hard rule on the *line* level.
- **Recommended action:** Add an explicit invariant to `docs/60-accounting/README.md`: *every posted line carries the Equipment code.* This is the hook downstream reports use to build the per-vehicle view (point 11).

### Point 3 — Add four new JobTypes; introduce a JobType taxonomy

- **Current spec:** `docs/30-domain-model/job-card.md` describes Job Cards as records of work — repair, sale preparation, warranty — but does **not** define a typed *JobType* category attribute on the Job Card.
- **Expert input:** Six JobTypes total (2 existing + 4 new): Reparation, Garanti, Toldsyn, Udlæg for nummerplader, Udlæg for registreringsafgift, Eksportgodtgørelse.
- **Verdict:** *fills a gap* — introduces a categorisation concept the current model is silent on.
- **Recommended action:** Add a "JobType" subsection to `docs/30-domain-model/job-card.md` listing the six categories with one-line definitions and their effect on equipment value / accounting. Treat JobType as a required attribute on every Job Card.

### Point 4 — Three new ancillary SKUs

- **Current spec:** `docs/30-domain-model/product.md` currently describes two product families: Equipment Inventory Products (`BIL-LAGx`) and Equipment Sales Products (`BIL-xx`). It mentions ordinary Products for spare parts and labour but doesn't enumerate vehicle-trade ancillary SKUs.
- **Expert input:** Three SKUs — Toldsyn, *Udlæg for nummerplader*, *Udlæg for registreringsafgift* — each with a Product Segment mapping to a specific e-conomic account. No SKU for *Eksportgodtgørelse*.
- **Verdict:** *extends* — adds a third product family.
- **Recommended action:** Add a third product-family section to `docs/30-domain-model/product.md` (working title: *Vehicle-trade ancillary products*) with a table listing the three SKUs and their VAT treatment from point 6.

### Point 5 — Vehicle AddOn must let each JobType be selectable and account-mapped

- **Current spec:** *Vehicle AddOn* is not described in `docs/`. It appears to be an existing X2 platform concept that hosts the per-tenant configuration for the Vehicle Trade module.
- **Expert input:** The AddOn needs (a) JobType dropdown entries for the four new types, (b) configurable e-conomic account numbers per type (Toldsynkonto, Nummerpladekonto, Registreringsafgiftkonto, Eksportgodtgørelseskonto).
- **Verdict:** *fills a gap* — introduces a configuration surface the spec has not described.
- **Recommended action:** Either (a) add a brief *Vehicle AddOn configuration* section in `docs/60-accounting/README.md` capturing the per-JobType account mapping, or (b) acknowledge Vehicle AddOn as a platform concept and document the v2vt-relevant settings only. Recommendation: option (a) — keep it close to the accounting integration where it bites.

### Point 6 — VAT treatment of the three ancillary SKUs

- **Current spec:** Not present.
- **Expert input:** Toldsyn = 25 % (Moms); both *Udlæg* SKUs = 0 % (Ej moms).
- **Verdict:** *fills a gap*.
- **Recommended action:** In the new ancillary-products table in `docs/30-domain-model/product.md`, populate the VAT rate / VAT description columns directly from the expert's specification:

  | SKU                             | VAT rate | VAT description |
  | ------------------------------- | -------- | --------------- |
  | Toldsyn                         | 25 %     | Moms            |
  | Udlæg for nummerplader          | 0 %      | Ej moms         |
  | Udlæg for registreringsafgift   | 0 %      | Ej moms         |

### Point 7 — *Registreringsafgift* JobCard is bidirectional

- **Current spec:** `docs/30-domain-model/job-card.md` doesn't address signed values on a Job Card.
- **Expert input:** This specific JobType supports both positive (sale) and negative (purchase) values on the same card type.
- **Verdict:** *extends*.
- **Recommended action:** Note in the JobType taxonomy (point 3) that *Udlæg for registreringsafgift* allows both signs, with the semantic spelled out (positive = charged to buyer, negative = paid out at purchase).

### Point 8 — *Eksportgodtgørelse* (export refund) workflow

- **Current spec:** Not modelled. `docs/50-processes/README.md` has no export-refund process.
- **Expert input:** Multi-step lifecycle — estimate via Told & Skat calculator → record on a JobCard → wait months → update with actual value → close. Feature request: in-system link to the calculator.
- **Verdict:** *fills a gap*.
- **Recommended action:**
  1. Add a new process to `docs/50-processes/README.md`: *Apply for export refund (Eksportgodtgørelse)* with the estimate → actual update lifecycle.
  2. Add the Told & Skat calculator URL to `docs/90-references/README.md` under External integrations: `https://motorregister.skat.dk/dmr-kerne/registreringsafgift/beregnRegistreringsafgift?execution=e1s1`.
  3. Add the deep-link requirement to `docs/70-user-interface/README.md` (JobCard editor).

### Point 9 — Margin-scheme VAT formula with EU export-refund adjustment

- **Current spec:** `docs/40-purchase-types/README.md` Type 2 (Margin Scheme) names *brugtmomsordningen* and Momsloven §§69–71, says VAT is calculated "on the profit margin (sale price − total purchase cost)", but does **not** give the explicit `× 25 / 125` formula or address the export-refund adjustment.
- **Expert input:** Concrete formula `Diff moms = ((Sale − Refund) × 25 / 125) − (Cost × 25 / 125)` for EU export under margin scheme, with a **floor at zero** when posting to e-conomic.
- **Verdict:** *extends* with a specific calculation rule the spec needs to encode.
- **Recommended action:** Add a *Calculation* subsection to `docs/40-purchase-types/README.md` under Type 2 with the explicit formula, the EU-export adjustment, the worked example from the notes, and the floor-at-zero rule. Cross-reference from the accounting section so the posting code knows about the floor.

### Point 10 — Warning at invoice time for non-margin-scheme vehicles

- **Current spec:** `docs/70-user-interface/README.md` *Sell equipment* section is a placeholder, with no validation/warning requirements yet.
- **Expert input:** When the user invoices a vehicle not bought-and-sold under the margin scheme, show: *"Remember to add registreringsafgift on invoice if required."*
- **Verdict:** *fills a gap*.
- **Recommended action:** Add to `docs/70-user-interface/README.md` under *Sell equipment* a *Validations and warnings* sub-bullet capturing this specific prompt with its condition (purchase type ≠ Margin Scheme).

### Point 11 — Per-vehicle expense and income view

- **Current spec:** `docs/70-user-interface/README.md` *Review inventory* mentions per-vehicle history including "purchase document, repairs, current carrying value, days in stock, expected margin" — but written before JobType was a concept.
- **Expert input:** Once the new JobTypes exist, the same view should aggregate every cost and income across every JobType for a single vehicle. Workshops can decide the level of granularity.
- **Verdict:** *confirms* the goal, *extends* by requiring all JobTypes to be aggregated.
- **Recommended action:** Update `docs/70-user-interface/README.md` *Review inventory* to confirm the per-vehicle drill-down covers all JobTypes (not only Repair/Warranty) and that the level of granularity is a per-tenant setting.

## Cross-cutting

- **e-conomic is named as the target accounting system.** Current `docs/60-accounting/README.md` keeps the accounting integration abstract. The expert writes as though e-conomic is *the* target. Decision needed: is e-conomic the single first-class integration, the primary one (others later), or one of several? Whatever the answer, `docs/60-accounting/README.md` should be clear about it.
- **Platform concepts not yet in `docs/`** appear repeatedly: *JobType*, *Vehicle AddOn*, *Product Segment*. These are X2 inheritances. They need either short glossary entries or a "Platform concepts assumed" section near the top of the domain model — otherwise the spec reads as if Vehicle Trade introduces these terms, which it does not.
- **JobType taxonomy is the pivot.** Multiple expert points (3, 5, 6, 7, 8, 11) all hang off a typed JobType attribute on the Job Card. Adding that taxonomy upfront — even before the four new JobTypes are wired — should be the first edit, because everything else snaps onto it.
- **The expert's example uses inside-EU.** The non-EU export-refund case is not stated. The spec should either (a) state the same rule applies, or (b) flag this as a separate open question — but should not be silent.
