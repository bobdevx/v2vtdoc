# Analysis 101 — Trade-in (part-exchange) flow modelled on Danish OE systems

Source: [notes/101.md](../notes/101.md)

## Per-point findings

### Point 1 — Align with Danish OE systems

- **Current spec:** `docs/70-user-interface/README.md` section 5 (*Vehicle trade-in*) is a stub. `docs/50-processes/README.md` lists *Handling of part-exchange / trade-in as part of a sale* as an open question with no entity-level process defined.
- **Expert input:** Adopt the pattern used by Dracar / Keyloop, Vitec Datamann, and CAC — a single combined invoice that both sells the outgoing vehicle and takes the trade-in into stock.
- **Verdict:** *fills a gap*.
- **Recommended action:** Use the expert's pattern as the design baseline for the trade-in process and UI; document it explicitly in `docs/50-processes/README.md` and `docs/70-user-interface/README.md`.

### Point 2 — Reference systems

- **Current spec:** No mention of comparator systems.
- **Expert input:** Names Dracar / Keyloop, Vitec Datamann, CAC.
- **Verdict:** informational; useful provenance.
- **Recommended action:** Optionally note the comparator systems in `docs/90-references/README.md` under *Internal X2 references* (or a new *Comparator systems* item). No requirement change.

### Point 3 — Outgoing vehicle in stock

- **Current spec:** Already covered by the standard `Stock → Sold` transition in `docs/30-domain-model/equipment.md`.
- **Expert input:** Confirms — *"as we already do now"*.
- **Verdict:** *confirms*.
- **Recommended action:** No change.

### Point 4 — Customer's trade-in offer

- **Current spec:** The trade-in stub in `docs/70-user-interface/README.md` section 5 already frames the scenario as "a buyer trading their old vehicle in part-payment".
- **Expert input:** Same framing with a concrete example.
- **Verdict:** *confirms*.
- **Recommended action:** No change.

### Point 5 — Trade-in created as Draft first

- **Current spec:** `docs/30-domain-model/equipment.md` defines the `(new) → Draft` transition and says *"a draft purchase document is normally created alongside"*. That note assumes the standard purchase flow.
- **Expert input:** For a trade-in, the Draft equipment row is created **without** a separate draft purchase document — the sales invoice for the outgoing vehicle serves that purpose.
- **Verdict:** *extends* — exposes an exception to the existing wording.
- **Recommended action:** Soften the `(new) → Draft` row's constraint in `docs/30-domain-model/equipment.md` to acknowledge the trade-in exception (e.g., *"…unless the row will be finalised by a sales invoice as a trade-in"*).

### Point 6 — Sales invoice from the outgoing vehicle is the trigger

- **Current spec:** `docs/50-processes/README.md` documents *Sell equipment via sales invoice* as a standalone process — sells one vehicle, period. There is no combined sale + acquisition process.
- **Expert input:** The sales invoice for the outgoing vehicle is also the document that finalises the trade-in acquisition; both transitions are atomic on a single posting.
- **Verdict:** *extends*.
- **Recommended action:** Add a new process to `docs/50-processes/README.md` — *Trade-in (part-exchange) sale* — and explicitly state the atomicity. Update `docs/30-domain-model/equipment.md` to record that the `Draft → Stock` trigger for a trade-in is the sales invoice for the other vehicle (not the trade-in's own purchase document).

### Point 7 — UI affordance to attach the trade-in

- **Current spec:** `docs/70-user-interface/README.md` section 5 stub is generic.
- **Expert input:** A specific UI requirement — somewhere on the sale screen there must be an affordance to attach an existing Draft trade-in (tab, below the lines, or under an Actions menu). Placement is left open by the expert.
- **Verdict:** *extends*.
- **Recommended action:** Expand `docs/70-user-interface/README.md` section 5 to require the attach affordance, without fixing its placement (the spec should be platform-friendly).

### Point 8 — One posting does two things atomically

- **Current spec:** `Draft → Stock` and `Stock → Sold` are documented as independent transitions, each driven by its own document.
- **Expert input:** In the trade-in case, the sales-invoice posting causes both transitions atomically: outgoing → Sold AND trade-in → Stock.
- **Verdict:** *extends*.
- **Recommended action:** Document the atomicity in `docs/50-processes/README.md` under the new trade-in process. Cross-link to the relevant transition rows in `docs/30-domain-model/equipment.md`.

### Point 9 — Single document covers both vehicles

- **Current spec:** Not stated.
- **Expert input:** The invoice carries all information about the trade-in vehicle alongside the outgoing one.
- **Verdict:** *extends*.
- **Recommended action:** Add this requirement to `docs/70-user-interface/README.md` (the invoice rendering must include trade-in details) and reinforce in `docs/60-accounting/README.md` (single posting batch carries both equipment codes).

### Point 10 — Accounting postings

- **Current spec:** `docs/60-accounting/README.md` does not yet have an explicit trade-in posting pattern.
- **Expert input:** Two illustrative postings — trade-in to inventory (negative cash entry equal to the agreed trade-in value) and a corresponding deduction on the customer's invoice. Net cash receipt = sale price − trade-in value.
- **Verdict:** *fills a gap*.
- **Recommended action:** Add a *Trade-in (part-exchange)* posting pattern to `docs/60-accounting/README.md` describing both legs. Reinforce the per-vehicle line tagging invariant that already came in via [analysis 100](../analysis/100.md).

## Cross-cutting

- **Atomicity is the central design choice.** The whole flow rests on a single sales-invoice posting causing two equipment-status transitions and the corresponding ledger entries. The spec must make this explicit; downstream, this affects retry, idempotency, and rollback semantics in the accounting integration.
- **Trade-in purchase type is unstated.** The example is a private customer (margin scheme), but the spec should let the trade-in's purchase type be set (margin scheme for private, purchase invoice for business). UI should default sensibly based on the buyer's status.
- **Pricing.** The trade-in valuation is the lever the customer negotiates. Whether it's captured on the Draft row at creation or on the sale screen at attach-time is a UX decision that has accounting consequences (it determines the lot value at `Draft → Stock` time).
- **Reversal symmetry.** `docs/30-domain-model/equipment.md` already has a Reversal/cancellation table. A credited trade-in sale needs an explicit row — most likely the trade-in goes `Stock → Reverted`, mirroring the outgoing vehicle going `Sold → Stock`. Worth specifying to avoid divergent implementations.
- **Implicit recommendation:** "Trade-in" is a strong enough first-class concept that introducing a *Trade-in Equipment* sub-flavour in the equipment lifecycle (or a `is_trade_in` flag) may be worth considering, so reports and dashboards can distinguish standard purchases from part-exchange acquisitions.

## Open issues to fold into the spec

Copy-ready bullets grouped by target file.

**`docs/30-domain-model/equipment.md`**
- Note an **exception** to the `(new) → Draft` constraint: for a trade-in, no separate draft purchase document is created — the row is finalised by the sales invoice for the *other* vehicle.
- Add a corresponding **trade-in trigger** row to the `Draft → Stock` description: *"…or, for a trade-in, posting the sales invoice that attaches this row as part-exchange."*
- Add a Reversal/cancellation row for the trade-in: when the linked sales invoice is credited, the trade-in equipment goes `Stock → Reverted` (to confirm).

**`docs/50-processes/README.md`**
- Replace the existing open question (*Handling of part-exchange / trade-in as part of a sale*) with an explicit process — **Trade-in (part-exchange) sale**. Sequence:
  1. The trade-in vehicle is created as a `Draft` Equipment row, capturing the agreed valuation and the trade-in vehicle's identity.
  2. A sales invoice is initiated against the outgoing `Stock` Equipment.
  3. The trade-in `Draft` is **attached** to the sales invoice via the UI affordance.
  4. The sales invoice is posted. In one atomic operation: outgoing vehicle goes `Stock → Sold`, the trade-in goes `Draft → Stock` (lot created at the agreed value, inventory aggregate incremented), and the trade-in value is deducted from the cash amount due from the customer.
  5. The invoice document carries all information for both vehicles.

**`docs/70-user-interface/README.md`**
- Expand section 5 *Vehicle trade-in* with the concrete UX:
  - Before the sale: create the trade-in as a `Draft` (capture VIN, registration, agreed value).
  - On the sale screen for the outgoing vehicle: a UI affordance (placement TBD — tab, section below the lines, or Actions menu) to attach an existing Draft trade-in to the current sale.
  - Posting: a single invoice action commits both sides; the rendered invoice carries trade-in vehicle details alongside the outgoing one.
- Reference systems whose pattern is being followed: Dracar / Keyloop, Vitec Datamann, CAC.

**`docs/60-accounting/README.md`**
- Add a posting pattern — **Trade-in (part-exchange)**:
  - Trade-in vehicle added to inventory at the agreed value (cashbook leg).
  - Customer invoice: trade-in value deducted from the cash amount due.
  - Net cash receipt = sale price − trade-in value.
  - Both legs share the same posting batch and carry both equipment codes (extends the per-line equipment-code invariant from [analysis 100](../analysis/100.md)).

**`docs/20-glossary/README.md`**
- Add: **Trade-in / part-exchange** — a vehicle taken from a buyer in part-payment for one of the workshop's stock vehicles. Created as `Draft`, attached to the sales invoice of the outgoing vehicle, taken into stock atomically with that posting.

**`docs/90-references/README.md`**
